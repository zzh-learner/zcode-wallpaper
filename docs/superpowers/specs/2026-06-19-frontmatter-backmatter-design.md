# 设计：前言 + 后记/番外识别

**日期**：2026-06-19
**子系统**：小说阅读器（`lib/reader-toc.cjs` 切章逻辑）
**关联**：AGENTS.md「小说阅读器」已知遗留 #1「最后一章吞后记/番外」

## 背景（为什么要做）

`parseTOC` 末章 `endOffset = text.length`，没有"第X章"标题的内容（后记、番外、网站声明、作者感言）全被并进最后一章。实测样本里一本小说最后一章 **29278 段 / 147 万字**——明显是后记堆出来的怪物章节，阅读体验崩。

用户需求（原话）：「能不能像章节识别那样，识别成后记或者番外，此外还有前言也是，可以把开头第一章前面的都算做前言。」

## 事实依据（6 本真实样本探测，教训 19）

探测脚本：`scripts/inspect-frontback.cjs`、`scripts/inspect-titles.cjs`。样本：`novels/` 下 6 本（回到明朝当王爷、天擎、惟我独仙、斗破苍穹、盘龙、纨绔才子）。

### 开头（第一个真章节之前）

| 书 | 第一非空行 | 开头标记词 |
|---|---|---|
| 回到明朝 | `※※※` 站点广告 | `书籍介绍` |
| 天擎 | （无） | （无） |
| 惟我独仙 | `《惟我独仙》` | `楔子` |
| 斗破苍穹 | `《斗破苍穹》` | （无） |
| 盘龙 | （无） | （无） |
| 纨绔才子 | `纨绔才子` | （无） |

**结论**：开头**没有稳定标题词**，6 本里只有 2 本有（"书籍介绍"、"楔子"），其余只有书名行 + 简介正文。所以前言识别**靠位置边界，不靠标题词**：第一个"第X章/节/回"标题之前的内容，归为一个前言章节。

### 结尾（最后一章正文里）

| 书 | 最后一章 | 后记标记词（在末章正文里的位置） |
|---|---|---|
| 回到明朝 | 第一章。（注：parseTOC 误识，见下） | **（无）**，只有文件末尾 `※※※` 站点广告 |
| 天擎 | 第二十二节 让我们的生命怒放 | `（全文完）`@6303 |
| 惟我独仙 | 第224章惟我独仙（终章） | `尾声`@6246、`（全书完）`@8622 |
| 斗破苍穹 | 第一千六百四十八章 …（大结局） | （无）—— 大结局就是正文结尾 |
| 盘龙 | 第四十三章 …（大结局）（下） | `全书终`@9530 |
| 纨绔才子 | 第九十七节 花落花开(大结局) | `（全文终）`@8461 |

**结论**：后记有**稳定标记词**（`尾声`/`（全文完）`/`（全书完）`/`（全文终）`/`全书终`/`后记`/`感言`/`番外`），5 本里有明确标记。1 本（回到明朝）**无标记词**，是边界情况（见下）。

## 设计

### 1. 前言识别（第一章之前）

**逻辑**：
1. 找到第一个"真章节"（`parseTOC` 现有 CHAPHEAD_RE 匹配的第一个章节）的 `startOffset`。
2. `beforeText = text.slice(0, firstCh.startOffset)`。
3. **过短不生成**：`beforeText` 非空行数 < 5 **或** 字符数 < 200 → 不生成前言章节（盘龙、天擎是 0 行，正合适不生成）。
4. 否则生成前言章节：`{title, startOffset: 0, endOffset: firstCh.startOffset}`，插入 `chapters` 数组最前面。

**标题词识别**（用户决策："用原文标题词"，回退"前言"）：
按优先级在 `beforeText` 里找标记词，第一个命中的作为标题：
`楔子` / `序章` / `序言` / `序` / `引子` / `引言` / `简介` / `书籍介绍` / `内容简介` / `作品相关` / `写在前面`

都匹配不到 → 标题 = `"前言"`。

**注意**：`序` 单字会误匹（如"序号""程序"）。规则要求 `序` 作为**独立行或行首词**才认（`\b序\b` 中文用行首 `^序\s*$` 或 `^序[：: ]`）。其他多字词直接 `includes` 即可。

### 2. 后记/番外识别（最后一章之后）

**逻辑**：
1. 取最后一章 `lastCh`，`lastBody = text.slice(lastCh.startOffset)`。
2. 从 `lastBody` 里**从前往后**找第一个后记标记词的位置（用第一个，因为"尾声"在"（全文完）"之前，是真正的内容分界）。
3. 找到则切分：
   - `lastCh.endOffset = lastCh.startOffset + markerIndex`（末章截断到标记词前）
   - 新增后记章节：`{title, startOffset: lastCh.startOffset + markerIndex, endOffset: text.length}`，push 到 `chapters` 末尾。
4. 找不到 → 不切分，保持现状（末章 `endOffset = text.length`）。

**标记词**（按优先级，决定标题）：
- `尾声` → 标题 `"尾声"`
- `番外` → 标题 `"番外"`
- `后记` / `完本感言` / `完结感言` / `感言` → 标题 `"后记"`
- `（全文完）` / `(全文完)` / `（全书完）` / `(全书完)` / `（全文终）` / `(全文终)` / `全书终` / `全文完` / `全文终` / `全书完` → 标题 `"后记"`（"完本声明"类，没有更具体的标题词，统一叫后记）

**匹配要求**：标记词要**独立成行或位于行首**才认，避免正文里"全文完成XX任务"误匹（教训 20：守卫代价权衡）。具体：标记词所在行 trim 后等于标记词本身，或以标记词开头且后跟标点（`。`/`！`/`）`/换行）。

### 3. 边界情况（回到明朝型，用户决策 A）

**最后一章正文里没有任何后记标记词**（回到明朝当王爷）。无法可靠区分"正文结尾"和"作者感言"，强切会误伤正文（教训 20）。

**决策**：**不强制切分，记为已知遗留**。6 本里 1 本受影响。在 `readertoctest.cjs` 用例记录此权衡，在 AGENTS.md 已知遗留更新。

### 4. 数据结构（不变）

`parseTOC` 返回的 `chapters` 数组元素结构 `{title, startOffset, endOffset}` 不变，只是数组前后可能各多一个元素。前端 `reader/reader.js`、`reader/book.js` **不需要改**——本来就按 chapters 数组渲染，多两个章节自然显示。

### 5. 实现位置

全部改在 `lib/reader-toc.cjs` 的 `parseTOC` 函数里（现有函数末尾、return 前加分流逻辑）。**不新增文件**，不改 `reader-server.cjs`、`reader/lib/toc.js`（前端 file: 模式的兜底 TOC 用的是简单 fallback，不识别前后记，保持现状——file: 模式本来就只读拖进来的 txt，前后记识别收益低）。

> **注意**：`reader/lib/toc.js`（前端）和 `lib/reader-toc.cjs`（server）是**两份独立实现**（跨环境无法共享代码，AGENTS.md 教训 17）。本次只改 server 端。file: 模式拖拽兜底用的是前端的简易 TOC，不做前后记识别（YAGNI——拖进来的文件 server 没扫过，识别收益低，且 file: 模式是退化路径）。

## 不做的事（YAGNI）

- 不识别"楔子"作为独立卷（惟我独仙的"第一集楔子"被 VOLHEAD 当卷标题的边角——不影响前言识别，前言只看"第一个章标题之前"）。
- 不做"番外集"子结构（多篇番外合并成一个"番外"章节，不展开）。
- 不改前端 `reader/lib/toc.js`（file: 兜底模式不做前后记识别）。
- 不动 server 端口/重连/编码任何现有逻辑。
- 不处理"回到明朝型"无标记词的后记（边界情况 A，记为遗留）。

## 已知遗留（本设计后）

1. **无标记词的后记无法切分**（回到明朝型）。文件末尾只有站点广告或纯感言、没有任何 `尾声`/`（全文完）` 类标记词时，后记仍会并进最后一章。无法可靠区分正文与感言，强切误伤风险高于收益（教训 20）。
2. **`序` 单字误匹残留风险**：靠"独立成行/行首"守卫，但极罕见的正文里以"序"开头成行的内容仍可能误识为前言标题。低概率，接受。
